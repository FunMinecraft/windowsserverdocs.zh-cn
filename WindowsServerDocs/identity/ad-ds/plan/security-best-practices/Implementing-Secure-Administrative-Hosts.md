# <a name="implementing-secure-administrative-hosts"></a>实现安全管理主机

>适用于：Windows Server 2016，Windows Server 2012 R2、Windows Server 2012

安全管理主机是工作站或已专为域控制器、加入域的系统，以及加入域的系统上运行的应用程序创建从中特权的帐户可以执行的管理任务 Active Directory 或安全平台的用途配置服务器。 在此情况下，"特权的帐户"是指不仅帐户的 Active Directory 中，但到已的任何帐户特权最大的组成员委派权利以及允许管理要执行的任务的权限。  
  
这些帐户可能具备重置密码的大部分域中的用户，用于管理区域和 DNS 记录的帐户或用于配置管理的帐户的帮助 Desk 帐户。 安全管理主机致力于管理功能，并 Microsoft Office 等这些不会运行如电子邮件应用程序、web 浏览器或生产力软件的软件。  
  
尽管"特权最大"的帐户和组相应地应严格最受保护，这不会消除需要保护的任何帐户和的权限上方这些标准用户帐户有权的组。  
  
安全管理主机可以仅用于管理任务，成员服务器种运行远程桌面网关服务器角色并 IT 用户连接来执行目标主机或运行 Hyper-V 角色，并提供每个 IT 用户使用他们的管理任务的唯一的虚拟机的服务器管理专用的工作站。 在许多环境中，可以实现组合所有三种方法。  
  
实现安全管理主机需要规划和与你的组织的大小、管理实践、风险兴趣和 budget 一致的配置。 注意事项和实现安全管理主机的选项供你在开发适用于你的组织管理战略使用下面提供。  
  
## <a name="principles-for-creating-secure-administrative-hosts"></a>用于创建原则安全管理主机  
实际上安全攻击的系统，应记住保持几常规原则：  
  
1.  你永远不会从受信任的不太主机（同样的角度，它管理在系统为不安全工作站）管理受信任的系统（即安全如域控制器服务器）。  
  
2.  你应该不依赖于单个身份验证因素时执行特权的活动;也就是说，用户的名称和密码组合不应该将视为接受身份验证由于表示仅单因素（知道你的某些内容）。 你应该考虑生成和缓存或管理方案中存储的凭据。  
  
3.  尽管大多数攻击当前威胁发展利用恶意软件以及恶意黑客，但不要跳过物理时设计和实现安全管理主机的安全。  
  
### <a name="account-configuration"></a>帐户配置  
即使你的组织当前未使用智能卡，你应该考虑实现它们特权的帐户和管理主机的安全。 需要智能卡登录的所有帐户链接到包含管理主机华丽绚烂的 GPO 中将以下设置配置管理主机：  
  
**计算机配置 \ 策略 \windows 本地安全 Options\Interactive 登录：需要智能卡**  
  
此设置需要所有交互式登录使用智能卡，无论 Active Directory 中将个人帐户上的配置。  
  
您还应该配置安全管理主机，以允许登录仅通过授权的帐户，可以用来配置：  
  
**计算机配置 \ 策略 \windows 本地安全本地策略用户版权分配**  
  
此授予交互式（以及，在适当远程桌面服务）的安全管理主机的授权的用户只登录权限。  
  
### <a name="physical-security"></a>物理安全  
被认为是可信的管理主机，他们必须配置并保护他们管理系统相同的程度。 建议中提供的大多数[针对攻击保护域控制器](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md)也有适用于用于管理域控制器和广告 DS 数据库主机。 在大多数环境中实现管理的安全系统的难题是物理安全可能更难以实现，因为这些计算机通常位于不可尽可能安全服务器中心，如管理用户桌面托管的领域。  
  
物理安全包含控制物理管理主机访问。 在较小的组织中，这可能意味着你维护专门的管理工作站保持锁定在 office 应用商店或桌子的抽屉里在不使用时。 或者，你需要执行的 Active Directory 或域控制器管理，当你登录到域控制器直接，这可能意味着。  
  
中等规模的组织中可能考虑实现安全管理"跳转服务器"，位于 office 在安全位置时的 Active Directory 或域控制器管理，则需要, 使用。 你也可以实现在安全位置，当不在使用，没有跳转服务器或锁定的管理工作站。  
  
在较大的组织中，你可以将提供对 Active Directory; 严格控制访问权限的数据中心存放跳转服务器部署域控制器;和文件、打印、或应用程序的服务器。 跳转服务器体系结构实现是最有可能包括较大的环境中的安全工作站和服务器的组合。  
  
无论为你的组织的大小和管理您的主机设计，你应该安全物理计算机免受未经授权的访问或被盗，并应使用 BitLocker 驱动加密加密并保护上管理主机的驱动器。 通过实现 BitLocker 管理主机上时，即使主机被盗或其磁盘将被删除，你可以确保驱动器上的数据是未经授权的用户都无法访问。  
  
### <a name="operating-system-versions-and-configuration"></a>操作系统版本和配置  
管理的所有主机，是否服务器或工作站，应运行最新的操作系统在使用你的组织中本文档之前所述的原因。 通过从新的安全功能、完整供应商支持和其他功能操作系统中引入了运行当前操作系统、管理人员权益。 此外，当您要将其部署到管理主机的第一次由评估新的操作系统，你将需要熟悉的新功能、设置和管理机制它提供，这随后可以利用在计划的操作系统的范围更广部署。 那时，你的组织中的最完善的用户也可以不熟悉新的操作系统和最佳定位为支持它的用户。  
  
### <a name="microsoft-security-configuration-wizard"></a>Microsoft 安全配置向导  
如果实现跳转服务器管理主机战略的一部分，你应使用内置的安全配置向导配置服务、注册表、审核，以及为了减少服务器攻击 surface 的防火墙设置。 当已收集并配置安全配置向导配置设置时，设置可以转换为 GPO 用于实施一致基准配置所有跳转服务器上。 你可以进一步编辑 GPO 以实现安全设置特定跳服务器，并可以结合使用其他基准设置解压缩从 Microsoft 安全合规性管理器中的设置的所有。  
  
### <a name="microsoft-security-compliance-manager"></a>Microsoft 安全合规性管理器  
[Microsoft 安全合规性管理器](https://technet.microsoft.com/library/cc677002.aspx)免费提供的工具集成安全配置 Microsoft，根据操作系统版本和角色配置，建议和一个的工具和可用于创建和配置基线域控制器安全设置的用户界面中收集它们。 Microsoft 可以与安全配置向导设置，以生成的跳转服务器的部署和部署在华丽绚烂的跳转的服务器位于 Active Directory Gpo 由实施的完整配置基线组合安全合规性管理器模板。  
  
> [!NOTE]  
> 此撰写时，Microsoft 安全合规性管理器中不包括特定于跳转服务器或其他安全管理主机的设置，但安全合规性管理器 (SCM) 仍可用于创建你管理主机的初始基准。 但是，若要正确安全主机，应该应用适用于高度安全的工作站和服务器的其他安全设置。  
  
### <a name="applocker"></a>AppLocker  
管理主机和虚拟 machinesshould 配置脚本、工具和应用程序白名单通过 AppLocker 或第三方应用程序限制软件。 应升级任何管理应用程序或不安全设置遵循的实用工具或替换为遵守安全开发与管理实践的工具。 当新的或其他工具需要管理主机上时，应全面测试应用程序和实用工具，并是否适用于管理主机上的部署工具，它可以将它添加到系统的白名单。  
  
### <a name="rdp-restrictions"></a>RDP 限制  
特定配置将会有所不同，具体取决于您管理的系统的体系结构，但你应包括的限制的帐户和计算机可用建立远程桌面协议 (RDP) 连接到托管的系统，如使用远程桌面网关（远程桌面网关）跳转服务器来控制访问域控制器和其他管理从授权的用户和新系统的系统。  
  
你应该允许交互式登录的授权的用户和应该删除或甚至阻止不需要的服务器访问其他登录类型。  
  
### <a name="patch-and-configuration-management"></a>修补程序和配置管理  
产品/服务，如 Windows 更新可能需要使用较小的公司或[Windows Server Update Services](https://technet.microsoft.com/windowsserver/bb332157) (WSUS) 管理部署的 Windows 系统更新，而较大的组织可能实现企业修补程序和配置管理软件，例如 System Center Configuration Manager。 无论你使用部署到你的常规服务器和工作站人口更新机制，您应该考虑单独部署高度安全如域控制器，证书颁发机构管理主机的系统。 按分离从常规管理基础结构这些系统，管理软件或服务帐户受到威胁，如果危害不能轻松扩展到最安全的系统，在您的基础结构。  
  
虽然不应实现的安全系统手动更新过程，你应配置更新的安全系统一个单独的基础结构。 即使在很大组织，该基础架构可以通常实现通过专用的 WSUS 服务器和 Gpo 的安全系统。  
  
### <a name="blocking-internet-access"></a>阻止的 Internet 访问权限  
不应允许管理主机访问 Internet 时，也不应他们将能够浏览公司的 intranet。 不应管理主机上允许 web 浏览器和类似的应用程序。 你可以通过 WFAS 配置，并且"黑洞"在安全的主机上的代理配置和外围防火墙设置组成的安全主机阻止 Internet 访问权限。 你还可以使用应用程序白防止管理的主机上使用 web 浏览器。  
  
### <a name="virtualization"></a>虚拟化  
如果可能，请考虑实现为管理主机的虚拟机。 使用虚拟化技术，你可以创建每位用户管理系统和的集中存储和管理，这可以轻松地在不使用时下，在关闭确保该凭据将不会保留活动管理系统上。 你还可以要求，虚拟管理主机将重置为最初的快照后每次使用，确保虚拟机保持原生态。 在以下部分中提供有关选项的虚拟化管理主机的详细信息。  
  
## <a name="sample-approaches-to-implementing-secure-administrative-hosts"></a>示例方法实现安全管理主机  
如何设计和部署您管理主机的基础结构，无论你应记住之前本主题中的"原则的创建安全管理主机"中提供的指南。 下面介绍的方法的每个提供有关如何你可以单独的"管理"和"工作效率"系统由 IT 人员的一般信息。 工作效率系统是 IT 管理员采用检查电子邮件、浏览 Internet 和使用 Microsoft Office 等常规生产力软件的计算机。 管理系统是强化和专门用于日常管理 IT 环境中的计算机。  
  
要实现安全管理主机的最简单方法是 IT 人员提供安全工作站，他们可以执行的管理任务。 仅工作站实现中, 每个管理工作站用于启动管理工具和 RDP 连接管理服务器和其他基础结构。 尽管变得更大，更复杂的基础结构可能会受益于分布式设计管理主机在其中使用专用管理服务器和工作站，述"实现安全管理工作站和跳转服务器"按照本主题后面，仅工作站实现可以在较小的组织中有效。  
  
### <a name="implementing-separate-physical-workstations"></a>实现单独物理工作站  
您可以实现管理主机的方法之一是处理 IT 的每个用户两台工作站。 一个工作站的"常规"用户帐户用于执行检查电子邮件和第二个工作站严格专用于管理功能时，使用生产力的应用程序等活动。  
  
为生产力工作站，可获得 IT 人员常规用户帐户，而不是无需使用特权的帐户登录到计算机不安全。 应使用严格控制配置配置管理工作站和 IT 人员应使用不同的帐户登录到管理工作站上。  
  
如果你已实现智能卡，管理工作站应该会配置为需要智能卡登录时，和 IT 人员应用于管理，还配置为需要智能卡登录提供单独的帐户。 管理主机应强化如上文所述，并且可以在本地登录到管理工作站指定的 IT 用户。  
  
#### <a name="pros"></a>优点  
通过实现单独物理系统，你可以确保将为其角色正确配置每台计算机和 IT 用户无意中不能公开管理系统风险。  
  
#### <a name="cons"></a>缺点  
  
-   实现单独物理计算机增加硬件成本。  
  
-   在登录到一台用于管理远程系统的凭据物理计算机缓存的内存中的凭据。  
  
-   如果管理工作站不安全地存储，它们可能易受攻击危害通过机制如物理硬件按键或其他物理攻击。  
  
### <a name="implementing-a-secure-physical-workstation-with-a-virtualized-productivity-workstation"></a>实现虚拟工作效率工作站安全物理的工作站  
在这种方法，IT 为用户提供安全管理工作站他们可以从中执行日常管理功能，负责的范围内他们使用远程服务器管理工具 (RSAT) 或 RDP 连接到服务器。 当 IT 用户需要执行的任务工作效率时，它们可到远程生产力工作站虚拟机的运行连接通过 RDP。 对于每个工作站，应使用单独的凭据，应该执行如智能卡的控制。  
  
#### <a name="pros"></a>优点  
  
-   管理工作站和工作效率工作站分隔。  
  
-   使用安全工作站连接到工作效率工作站上 IT 人员可以使用单独的凭据和智能卡和特权的凭据不放不太安全的计算机上。  
  
#### <a name="cons"></a>缺点  
  
-   实现该解决方案需要设计和实现工作且可靠的虚拟化选项。  
  
-   如果不安全地存储物理工作站，它们可能易受影响的硬件或操作系统，并使其更容易受到通信拦截物理攻击。  
  
### <a name="implementing-a-single-secure-workstation-with-connections-to-separate-productivity-and-administrative-virtual-machines"></a>实现单个安全的连接"生产力"和"管理"的虚拟机的分隔工作站  
在这种方法，你可以发出 IT 用户单个物理工作站，处于锁定状态如上文所述，并且在其上 IT 用户没有授权的访问。 你可以提供远程桌面服务连接到专用的服务器上托管了虚拟机 IT 人员提供了一台运行电子邮件和其他生产力的应用程序的虚拟机和配置为用户的专用管理主机的第二个虚拟机。  
  
你应该需要智能卡或其他因素登录的虚拟机，使用不同的帐户，而不是用于登录到物理计算机上的帐户。 IT 用户登录到物理计算机后，他们可以使用自己的工作效率智能卡连接到他们的办公远程计算机单独的帐户和智能卡连接到远程计算机管理。  
  
#### <a name="pros"></a>优点  
  
-   IT 用户可以使用单个物理工作站。  
  
-   按虚拟主机和使用远程桌面服务连接到虚拟机需要单独的帐户，不会在本地计算机上的内存中缓存 IT 用户的凭据。  
  
-   管理主机，减少本地计算机危害的可能性相同的程度，就可以保护物理主机。  
  
-   在其中 IT 用户办公虚拟机或其管理虚拟机可能已受到威胁的情况下，在虚拟机可以轻松地将重置"已知的良好"状态。  
  
-   物理计算机受到威胁后，如果没有特权的凭据将缓存在内存中，并使用智能卡可以阻止危害由击键记录器的凭据。  
  
#### <a name="cons"></a>缺点  
  
-   实现该解决方案需要设计和实现工作且可靠的虚拟化选项。  
  
-   如果不安全地存储物理工作站，它们可能易受影响的硬件或操作系统，并使其更容易受到通信拦截物理攻击。  
  
### <a name="implementing-secure-administrative-workstations-and-jump-servers"></a>实现安全管理工作站和跳转服务器  
替代进行保护管理工作站，或与它们结合使用，你就可以实现安全跳转服务器，并且管理用户可以连接到使用 RDP 和智能卡执行管理任务跳转服务器。  
  
跳转服务器应配置运行的远程桌面网关角色，以便您可以实现在连接到跳转服务器和将其从管理的目标服务器上的限制。 如果可能，应还安装 Hyper-V 角色，并创建[个人虚拟桌面](https://technet.microsoft.com/library/dd759174.aspx)或管理用户使用他们任务跳转服务器上的其他每位用户虚拟机。  
  
通过提供跳转服务器上每位用户虚拟机管理用户，你会提供管理工作站物理安全和管理用户可以重置或关机时不在使用其虚拟机。 如果你不希望在同一管理主机上安装了 Hyper-V 角色和远程桌面网关角色，可以在单独的计算机上安装它们。  
  
尽可能，应使用远程管理工具来管理服务器。 远程服务器管理工具 (RSAT) 功能应用户的虚拟机（或如果你未实现管理每个用户虚拟机的跳转服务器）上安装和管理人员应 RDP 通过连接到其虚拟机，若要执行的管理任务。  
  
在情况下时管理用户必须 RDP 通过连接到目标服务器来进行管理直接、远程桌面网关应将配置要允许在适当的用户和计算机用于建立连接目标服务器时，才会进行连接。 执行工具 RSAT（或类似）应未指定管理系统的系统上诸如常规使用工作站和的成员服务器不跳服务器禁止。  
  
#### <a name="pros"></a>优点  
  
-   创建跳转服务器可以将映射到"区域"（集锦的系统配置，连接和安全要求，类似）您网络中的特定服务器，并要求管理每个区域的通过管理连接到指定为"区域"服务器的安全管理主机的人员。  
  
-   通过将跳转服务器映射到区域，可实现连接属性和配置要求精确控制，并可轻松地找出尝试连接的未经授权的系统。  
  
-   通过实现跳转服务器上的每个管理员虚拟机，你将强制关机并重完成的管理任务的已知全新状态虚拟机重置。 通过强制执行虚拟机关机（或重启）管理任务完成后，虚拟机无法针对攻击者，也不是可行的凭据盗窃攻击，因为内存缓存凭据不会保留超出重新启动。  
  
#### <a name="cons"></a>缺点  
  
-   专用的服务器服务器都需要跳转，无论物理还是虚拟。  
  
-   实现命名跳转服务器和管理工作站需要仔细地计划和映射到配置环境中的任何安全区域的配置。

# <a name="create-a-new-nic-team-on-a-host-computer-or-vm"></a>在主计算机或 VM 中创建新的 NIC 团队

>适用于：Windows Server（半年通道），Windows Server 2016

本主题提供 NIC 组合配置的信息，以便了解新的 NIC 团队配置时，你必须进行选择。 本主题包含以下部分。  
  
-   [选择分组模式](#bkmk_teaming)  
  
-   [选择负载平衡模式](#bkmk_lb)  
  
-   [选择待机适配器设置](#bkmk_standby)  
  
-   [使用团队主要的界面属性](#bkmk_primary)  
  
> [!NOTE]  
> 如果您已了解这些配置商品，你可以使用下面的过程配置 NIC 组合。  
>   
> -   [在 VM 中创建新的 NIC 团队](../../technologies/nic-teaming/Create-a-New-NIC-Team-in-a-VM.md)  
> -   [创建新的 NIC 团队](../../technologies/nic-teaming/Create-a-New-NIC-Team.md)  
  
当你创建新的 NIC 团队时，你必须配置以下 NIC 团队属性。  
  
-   工作组名称  
  
-   成员适配器  
  
-   分组模式  
  
-   负载平衡模式  
  
-   待机适配器  
  
此外可以选择，您可以配置主要组接口和配置大量虚拟 LAN (VLAN)。

## <a name="bkmk_teaming"></a>选择分组模式  
适用于分组模式的选项是**切换独立**，**静态组队**，并**链接聚合控件协议 (LACP)**。 静态组队和 LACP 是依赖切换模式。 为了获得最佳 NIC 团队性能与所有三种分组模式，建议你使用的动态分配负载平衡模式。  
  
**切换独立型**  
  
带有独立切换模式、 交换机或交换机的成员连接 NIC 团队没有意识到 NIC 团队存在和不确定如何网络将通信分发给 NIC 团队成员-相反，NIC 团队分配站的网络流量跨 NIC 团队的成员。  
  
当你使用的动态分配的独立切换模式时，按动态负载平衡算法修改根据 TCP 端口地址哈希分发网络交通加载。 动态负载平衡算法重新分发优化团队成员带宽利用率以便个别流传输从一个活动的组成员可以移动到另的版本流。 将考虑，重新分发交通可能会导致无序传递数据包，以便它可以采取措施最小化该的可能性较小的可能性算法。  
  
**切换依赖于**  
  
依赖切换模式，切换到与它连接 NIC 团队成员确定分配 NIC 团队成员之间的入站的网络交通的方法。 切换具有完整独立来确定如何网络交通分布 NIC 团队的成员。  
  
> [!IMPORTANT]  
> 切换依赖分组要求团队中的所有成员都连接到相同的物理切换，或多机箱切换的共享多机箱之间切换 ID。  
  
静态分组需要你手动配置此开关，并找出其主机链接形成团队。 由于这是静态配置的解决方案，以帮助切换没有其他协议并且该主机不正确识别接通电源线或其他可能会导致无法执行团队的错误。 通过服务器类开关通常支持此模式。  
  
与不同的静态团队合作，LACP 组队模式动态标识连接主机和切换之间的链接。 此动态连接启用自动创建的团队，从理论上说但很少练习、 扩展和减少了团队中，只需通过传输或从等实体 LACP 数据包收到。 所有服务器类开关都支持 LACP，，并且所有都需要的网络运营商以管理上启用 LACP 上切换端口。 配置 LACP 分组模式时，NIC 组合始终以 LACP 的活动模式运行的简短计时器。  未选项，目前可修改计时器或更改 LACP 模式。  
  
当你通过动态分配使用了相关切换模式时，按动态负载平衡算法修改根据 TransportPorts 地址哈希分发网络交通加载。  动态负载平衡算法重新分发优化团队成员带宽利用率的版本流。 各个流传输可以一个活动的组成员之间移动作为动态分配的一部分。 将考虑，重新分发交通可能会导致无序传递数据包，以便它可以采取措施最小化该的可能性较小的可能性算法。  
  
为所有切换依赖配置切换都确定分配团队成员之间的入站的交通的方法。  预计切换合理好地分布团队成员的流量，但它具有完整独立来确定如何进行操作。  
  
## <a name="bkmk_lb"></a>选择负载平衡模式  
适用于负载平衡分配模式的选项是**地址哈希**， **HYPER-V 端口**，并**动态**。  
  
**地址哈希**  
  
此负载平衡模式中创建根据地址组件的数据包哈希代码。 然后，它将分配数据包具有可用适配器之一哈希该值。 通常会孤单此机制足以所有可用的适配器创建合理余额。  
  
你可以使用 Windows PowerShell 指定以下希函数组件的值。  
  
-   源和目标 TCP 端口和源代码和目的地的 IP 地址。 这是默认设置，当你选择**地址哈希**作为负载平衡这一模式。  
  
-   源和目的地的 IP 地址仅。  
  
-   源和目的地的 MAC 地址仅。  
  
TCP 端口哈希创建的交通流，导致在将独立移 NIC 团队成员之间的较小流最详细的 distribution。 但是，不能使用的 TCP 端口哈希不 TCP 的交通或 UDP 基于，或在 TCP 和 UDP 端口处于隐藏状态的堆栈向上，如 IPsec 保护流量。 在这些情况下，哈希自动使用的 IP 地址哈希或交通不 IP 通信，如果使用的 MAC 地址哈希。  
  
**Hyper V 端口**  
  
使用适用于配置 HYPER-V 主机的 NIC 团队 HYPER-V 端口模式处于利用。 由于虚拟机的功能有独立的 MAC 地址，VM 的 MAC 地址-或 VM HYPER-V 虚拟交换机用来在已连接到该端口-可以在其划分之间 NIC 团队成员网络通信的基础。  
  
> [!IMPORTANT]  
> 无法使用 HYPER-V 端口负载平衡模式配置虚拟机中创建的 NIC 团队。 使用地址哈希加载平衡模式。  
  
邻近切换始终看到一个端口上的特定的 MAC 地址，因为该开关将基于 MAC (VM MAC) 地址目的地的多个链接上分发入口加载 （的通信从切换到主机）。 这是非常有用时使用了虚拟机队列 (VMQs)，因为队列将其置于特定 NIC 需要到达通信。  
  
但是，如果主机仅少数虚拟机的功能，可能无法精细足够实现均衡 distribution 此模式。 此模式会还始终限制单独的 VM （即，单个切换端口的通信） 可用于单接口的带宽。 NIC 组合使用 HYPER-V 虚拟交换机用来端口，作为标识符，而不是因为在某些情况下，可能会使用多个切换端口上的 MAC 地址配置 VM 使用源 MAC 地址。  
  
**动态**  
  
此负载平衡模式可以利用另外两种模式的最佳方面，并将它们合并到单个模式：  
  
-   根据的 TCP 端口和 IP 地址哈希代码分发站加载。  动态模式还重新平衡实时加载，以便给定的站流可能团队成员之间来回移动。  
  
-   HYPER-V 端口模式的方式相同分发站的加载。  
  
在此模式中的站加载动态平衡根据 flowlets 的概念。 和人性语音有一样自然符在的字词和句子的末尾，TCP 流 （TCP 通信流） 也有自然众符。 部分之间此类两个符 TCP 流称为 flowlet。  
  
当动态模式算法发现，遇到 flowlet 边界-例如 TCP 流-中发生具有足够的长度休息一下算法自动重新平衡给其他团队成员流如果相应。  在某些情况下算法可能还会定期重新均衡它不包含任何 flowlets 的版本流。 出于此原因，之间 TCP 流和团队成员关系可以随时更改因为动态平衡算法起作用，平衡团队成员工作负载。  
  
是否团队配置了切换独立或之一依赖切换模式，建议你使用动态分配模式，以获得最佳性能。  
  
没有为此规则异常当 NIC 团队有两个团队的成员，配置在切换独立模式中，并已启用一个 NIC 活动的活动月待机模式和其他待机状态的配置。 使用此 NIC 团队配置，地址哈希 distribution distribution 提供比动态略有更好的性能。  
  
## <a name="bkmk_standby"></a>选择待机适配器设置  
待机适配器的选项是你选择的特定网络适配器，将作为一个待机适配器，虽然其他未选中的团队成员活动 NIC 团队或任何 （所有适配器活动）。 没有网络通信 NIC 配置为待机适配器时，已发送到或处理的适配器，除非和直到活动 NIC 失败时当前。 活动网卡无法正常工作后，待机 NIC 将处于活动状态，并进程网络通信。 服务还原所有团队的成员，在待机团队成员返回到待机状态。  
  
如果你有两个 NIC 团队，并且你选择要将一个 NIC 配置为待机适配器，你失去带宽活动的两个 nic 存在的聚合优势。  
  
> [!IMPORTANT]  
> 不需要指定实现故障功能; 待机适配器故障能力始终会出现在 NIC 团队至少两个网络适配器时。

# Windows Admin Center UI 文本和设计风格指南

>适用于： Windows Admin Center

本主题介绍编写 Windows Admin Center 用户界面 (UI) 文本的常规方法，以及我们采用的一些特定惯例和方法。

Windows Admin Center 和任何扩展名都应遵循 [Microsoft 语音原则](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)，以便提供易用、友好的体验。 此风格指南基于这些语音原则和 [Microsoft 编写风格指南](https://docs.microsoft.com/style-guide/welcome/)，因此请务必查看上述两种资源中有关[辅助工具](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements)、[缩略语](https://docs.microsoft.com/style-guide/acronyms)和[请](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please)、[抱歉](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)等[词汇选择](https://docs.microsoft.com/style-guide/word-choice/)的信息。

## 按钮

- 按钮名应尽可能使用一个单词，尤其是在打算本地化该工具的情况下。 两个或三个是确定，但请尽量避免较长时间。 如果你有四个字词或较长时间，它将是更好地使用链接控件。
- 按钮标签应该简洁具体而又明晰易懂。 使用与用户操作对应的动词，如“创建”、“删除”、“添加”、“格式化”等，而不是使用通用的“提交”按钮。
- 如果一个按钮后跟着一个问题，其标签应该清楚地回答该问题（通常为“是”或“否”）。

## 大写

我们遵循 Microsoft [大写](https://docs.microsoft.com/style-guide/capitalization)风格 - 基本全部使用句子元素的首字母大写。

| UI 元素              |大写|备注|
|-------------------------|--------------|--------|
|提醒标记（如 PREVIEW） |全部大写      ||
|其他所有内容          |首字母大写|但是，我们在 WMI 或 PowerShell 中展现对象属性时有一些例外情况是我们无法控制的。|

## 冒号

使用冒号引入列表。 例如：

    Choose one of the following:
    Cats
    Dogs
    Quokkas

当标签是在不同的行件事从其标签或标签和内容之间的明显区别时它标记时，不要使用冒号 UI 文本中。

标签位于同一行文本其标签并需要阻止在两个元素一起运行时，请使用冒号 UI 文本中。

## 确认消息

确认对话框非常有用，当将继续操作可能会有意外的结果，例如数据丢失。 他们应包含具有清晰的结果，特别是对于不可逆转的事件的易于、 有用信息。 

- 请确保确认是必需的。 如果不没有提供任何新信息 （例如，"确实吗？"） 然后可能不需要一封确认邮件。  
- 验证客户想要继续操作。
- 请确保不冗余的主要说明 （标题） 和说明文本 （正文）。
- 在标题中，定义可能的结果问题或有关将发生什么情况下一步的语句。 例如，"清除此驱动器上的所有数据？或者，"你是要清除你的所有数据"。
- 添加的正文中的详细信息。 如果存在一个变量，如有关更改，你的项目名称包含在此处。
- 包含简单问题 （或在标头中的正文中） 的框架两个操作按钮之间的明确选择。
- 对于复杂的选项，使用是/否鼓励仔细阅读的按钮。 对于简单的选项，使用按钮特定于该操作，如删除所有或取消。

## 首次运行体验

可以使用工具帮助第一次访问某一页面的用户入门。 这可以是：

- 空页面中的带有入门方法简短介绍的文本字符串，例如，“选择‘添加’以添加应用。”
- 帮助用户入门的控件链接，例如，“添加应用并开始”。
- 一段向用户展示入门方法的简短动画或视频

以下是我们 Windows 风格指南中的一些提示：

### 1. 有所帮助

- 避免使用营销风格或语言。
- 当演示或建议某事时，确保最终结果简洁明了；如果客户不知道这么操作的原因，那么只向他们展示操作方法是没有效果的。
- 不显示客户不需要的提示。

### 2. 展示而非讲述

尽可能简化文本（考虑小型动画或视频）。

### 3. 避免信息量过大

- 每个组合使用情况会话的弹出窗口和提示最多为 4 个，包括系统通知和 shell 通知。
- 确保弹出窗口的出现时机合适。
- 不要阻止客户执行操作。
- 确保弹出窗口可以轻松消除。

### 4. 符合情境

- 教学片段在恰当时间显示最为有效。
- 创建教程或幻灯片时需确保信息具体明确。
- 拒绝营销“垃圾”- 专注于具体提示和技巧。
- 如果有必要，为客户提供之后返回教程的方法（安装说明可能仅相关一次，而人们第一次通常不会保留信息）。
- 空页面消息应该能让用户了解消息出现的原因并且/或者让用户感到舒心，因此应确保消息简洁实用。

### 5. 减少费力安装

客户为了体验全部价值所需执行的其他操作（如登录到在线服务）应尽可能容易。

- 消息应该简短直接。
- 避免客户流失。 如果可能，请提供一种从客户所在地进行连接的方法。
- 如果可以，提供稍后执行的选项，然后提醒客户稍后执行。
- 如果在客户体验中略过了一些选项，应提供快速简单的切回方法。

## 帮助链接

以下是我们 Windows 风格指南中的一些提示：

### 何时我们应提供的帮助链接？

几乎永远不会。 提供的帮助链接时，才：

- 没有客户很可能具有的答案将帮助他们在 UI 任务时成功在 UI 中使用时明显和重要问题。 
- 若要提供需要用户在 UI 任务时成功的信息量在 UI 中没有足够的空间。 

### 其中应可以帮助显示的链接？ 

- 文本链接应显示的 UI 元素，帮助在引导尽可能接近。 
- 如果你必须提供适用于整个 UI 屏幕的文本链接，将其放置在屏幕的左下角。 
- 如果你提供通过帮助按钮 （？） 的链接，工具提示应为"帮助"。

### 我们应该使用哪些 URL？

永远不会直接链接到 web 地址 — 改为使用的重定向服务。

Microsoft 开发人员应使用除时，用户可能需要手动键入，哪些区分使用 aka.ms 链接 （只要 URL 的目标是自动识别浏览器区域设置，如 Docs.microsoft.com 网站） 的帮助链接 FWLink

### 文本指南 

- 使用完整的句子。
- 没有结束标点问号除外。 
- 你无需为任务主题作品; 使用相同的文本使用有意义的 UI，上下文中的文本，但应确保两个之间的逻辑连接。 例如： 
- 帮助链接： 允许例外风险是什么？ 
- 帮助主题作品:"允许通过 Windows 防火墙通信程序"
- 请尽可能具体有关帮助主题的内容。 
    - 我们样式
        - Windows 防火墙如何帮助保护我的电脑？
        - 为什么高光可以改进图片
    - 不是我们的样式
        - 有关 Windows 防火墙的详细信息
        - 了解有关颜色管理的详细信息
        - 了解详情
- 对于链接文本，而不只是这些关键字使用完整的句子。 
    - 我们样式 
        - [允许例外的风险有哪些？]()
    - 不是我们的样式
        - [允许例外的风险]()是什么？ 

## 错误消息

下面是 Windows 风格指南中进行调整以适应某些指南：

编写好消息是提供足够的说明，但不过于技术; 之间达成平衡之间的休闲和又备受青睐，但不是令人厌烦或者具有冒犯性。

### 一般指南

使用错误情况下每一条消息。

#### 标题

- 保持简短和简洁解释该问题是什么，或**理想情况下怎么办**。 <br>某些 UI 图面可能具有标题截断而不是环绕在他们完成太长，因此请留意这些。
- 如果它是一个简单的步骤，请在标题中使用的解决方案。
- 请确保在标题相关直接向该按钮，以防读卡器将忽略的正文文本。
- 避免使用"出现了问题"标题，除非你不有任何其他选项。 更为具体化有关该问题。
- 标题中，应避免使用变量 （例如文件、 文件夹和应用名称）。 将它们放入正文。

#### 正文

- 如果标题充分解释该问题或解决方案，你无需正文文本。
- 不要只是重复个措词消息中的标题。
- 清楚地通信和简洁的解决方案是什么。
- 侧重于首先使事宜。
- 不要错误责怪用户。
- 如果存在错误代码与错误相关联，如果你认为包括错误代码可能会帮助客户或 Microsoft 支持研究此问题，将其包括在正文文本下方，并将其写入，如下所示：

    错误代码: # # #

    如果客户有解决错误而无需代码所需的所有信息，你无需将其包括。

#### 按钮

- 编写按钮文本，以便它是对主要说明的具体响应。 如果这是不可能，请使用"关闭"解除按钮文本 （而不是"正常"或"完成"）。
- 如果你有多个按钮，将最左侧的按钮建议用户采取的操作。 使最右侧的按钮更保守的操作，如"取消。

#### 帮助链接

仅考虑你不能进行特定于且可操作的错误消息的帮助链接。

## Null 状态文本

下面是一些帮助从 Windows 风格指南。

客户数据或内容不存在时，会发生 null 状态从应用或功能中，搜索后，或者在需要时不返回任何结果信息时缺少表单，如计费交易记录信息。

### 指南

 - 如果可能，有机会使用 null 状态的情况下教育员工有关如何使用该功能 （例如，如何添加音乐，查找图片、 等。）  
- 如果你在 UI 中有一个主题作品，解释此操作需要"修复"null 状态 （例如，"添加某些音乐"） 
- 带有文本的乐趣。 此空间可以提供舒心，因为它可能不会看到几次机会。 
- 避免"这是孤独在这里。" 这是 sad，并且具有已过度使用。 
- 避免问题"未连接打印机？" 可以使用一次，但这种格式往往过度使用，并且问题将额外的负担压力置于客户。 它还可以随时 condescending。 
- Null 状态文本中的各种是一个好做法。 

### 示例

- "将某人添加为最喜爱的人物，并且你将在此处看到它们。"
- "明白任何成就或游戏剪辑便特别自豪？ 将它们添加到你展示。"
- "任何人的派对尚未。 启动一个 ！"
- "当某人将你添加为好友时，你将看到它们此处。"
- "时执行此操作像解锁成就，记录游戏剪辑，并添加好友，你将看到它这里。"
- "你最喜爱的好友这里会显示，以便你可以看到它们是在线和它们是最。"

## 标点符号

- 标题或不完整的句子不加结束标点符号（句号、问号）。 标题包含问题的确认对话框例外
- 使用《Microsoft 风格指南》中关于[句号](https://docs.microsoft.com/style-guide/punctuation/periods)和[问号](https://docs.microsoft.com/style-guide/punctuation/question-marks)的指南。

## 状态消息

状态消息由弹出 (toast) 消息和通知组成。

|字符串类型         | 注释                               |
|------------        |-------------------------------------|
|Toast               |带有结束标点符号且首字母大写 - 最好带有对象变量，以便用户能在离开对象后了解消息适用于哪个对象。|
|通知标题|没有结束标点符号且首字母大写（这是一个标题） - 最好带有对象变量|
|通知详情|完整句子，最好带有显示对象的 UI 链接|

下面是通知消息的一些详细建议：

|字符串类型         | 注释                               |
|------------        |-------------------------------------|
|已启动             |尽可能省略 - 通常可以直接跳到正在进行的消息以减少干扰次数。|
|正在进行         |开头是表示正在执行的操作的动词，结尾是表示操作正在进行的省略号。 下面是一个示例：<br> *正在创建“客户数据”卷...*|
|成功             |开头是“已成功”，结尾是软件刚刚完成的操作。 下面是一个示例：<br> *已成功创建“客户数据”卷。*|
|失败             |开头是“无法”，结尾是软件无法完成的操作。 下面是一个示例：<br> *无法创建“客户数据”卷。*|

## 工具提示

良好的工具提示简要描述没有标签的控件或标记控件，提供其他信息的位时这非常有用。 它们还可以帮助客户通过提供其他导航 UI — 不冗余-有关控件标签、 图标、 链接、 等信息。

尽量少或根本不应使用工具提示。 他们可以向客户，因此不包括只需重复标签或状态明显的工具提示中断。 它应始终添加有价值的信息。

|    上下文                                 |    如何编写工具提示    |
|    -----------------------                 |    -------------------------    |
|当控件或 UI 元素是未标记...|使用简单且具有描述性的名词短语。 例如：<br> 荧光笔 |
|当 UI 元素进行标记，但它的用途需要说明...|<ul><li>简要介绍如何使用此 UI 元素。 </li><li>使用强制性动词命令窗体。 例如，"文本中找到此文件"（不"文本中查找此文件"）。</li><li>不要包含结束标点符号，除非有多个完整的句子。</li> </ul>|
|文本标签时被截断或可能在某些语言中截断...|<ul><li>提供工具提示中指示的标签。</li><li>可选： 在另一行，阐明说明，但如果需要提供。</li><li>如果指示的信息页面或流上提供了其他地方，不提供工具提示。</li></ul>|
|键盘快捷方式不可用时...|<ul><li>可选： 提供括号中的标签或描述性的短语，例如"打印 (Ctrl + P)"或"查找此文件 (Ctrl + F) 中的文本"的键盘快捷方式</li><li>它是确定添加有用的键盘快捷方式阐明工具提示，但避免添加工具提示仅用于显示键盘快捷方式。 </li></ul>|

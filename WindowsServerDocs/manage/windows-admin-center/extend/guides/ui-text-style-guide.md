# <a name="windows-admin-center-ui-text-and-design-style-guide"></a>Windows Admin Center UI 文本和设计风格指南

>适用于：Windows Admin Center

本主题介绍编写 Windows Admin Center 用户界面 (UI) 文本的常规方法，以及我们采用的一些特定惯例和方法。

Windows Admin Center 和任何扩展名都应遵循 [Microsoft 语音原则](https://docs.microsoft.com/style-guide/brand-voice-above-all-simple-human)，以便提供易用、友好的体验。 此风格指南基于这些语音原则和 [Microsoft 编写风格指南](https://docs.microsoft.com/style-guide/welcome/)，因此请务必查看上述两种资源中有关[辅助工具](https://docs.microsoft.com/style-guide/accessibility/accessibility-guidelines-requirements)、[缩略语](https://docs.microsoft.com/style-guide/acronyms)和[请](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/p/please)、[抱歉](https://docs.microsoft.com/style-guide/a-z-word-list-term-collections/s/sorry)等[词汇选择](https://docs.microsoft.com/style-guide/word-choice/)的信息。

## <a name="buttons"></a>按钮

- 按钮名应尽可能使用一个单词，尤其是在打算本地化该工具的情况下。 两个或三个是确定，但请尽量避免再。 如果有四个字或更长，会更好，若要使用的链接控件。
- 按钮标签应该简洁具体而又明晰易懂。 使用与用户操作对应的动词，如“创建”、“删除”、“添加”、“格式化”等，而不是使用通用的“提交”按钮。
- 如果一个按钮后跟着一个问题，其标签应该清楚地回答该问题（通常为“是”或“否”）。

## <a name="capitalization"></a>大写

我们遵循 Microsoft [大写](https://docs.microsoft.com/style-guide/capitalization)风格 - 基本全部使用句子元素的首字母大写。

| UI 元素              |大写|备注|
|-------------------------|--------------|--------|
|提醒标记（如 PREVIEW） |全部大写      ||
|其他所有内容          |首字母大写|但是，我们在 WMI 或 PowerShell 中展现对象属性时有一些例外情况是我们无法控制的。|

## <a name="colons"></a>冒号

使用冒号引入列表。 例如：

    Choose one of the following:
    Cats
    Dogs
    Quokkas

标签是在不同的行的操作从其标签或标签和操作之间的明显区别时它标记时，不要在 UI 文本中使用冒号。

标签是文本的同一行上其标签和需要防止两个元素一起运行时在 UI 文本中使用冒号。

## <a name="confirmation-messages"></a>确认消息

确认对话框非常有用时继续操作可能具有意外的结果，例如数据丢失。 它们应包含与清除结果，尤其是对于不能颠倒的事件可以方式进行扫描、 有用信息。 

- 请确保一条确认信息是必需的。 如果不没有提供任何新信息 （例如，"确实吗？"） 则可能不需要一条确认消息。  
- 验证客户想要继续进行操作。
- 请确保主要说明 （标题） 和 （主体） 的解释性文本不冗余。
- 在标题中，为一个问题或有关接下来要发生的语句定义可能的结果。 例如，"清除此驱动器上的所有数据？或者"你将擦除所有数据"。
- 在正文中添加详细信息。 如果一个变量，例如有关更改，是的项的名称包括在此处。
- 包括简单问题 （不管是在标头或正文中） 的框架两个操作按钮之间的明确选择。
- 对于一个复杂的选择，使用是/否按钮，鼓励仔细阅读。 一个更简单的选择，使用按钮，是特定于操作，如删除所有或取消。

## <a name="first-run-experiences"></a>首次运行体验

可以使用工具帮助第一次访问某一页面的用户入门。 这可以是：

- 空页面中的带有入门方法简短介绍的文本字符串，例如，“选择‘添加’以添加应用。”
- 帮助用户入门的控件链接，例如，“添加应用并开始”。
- 一段向用户展示入门方法的简短动画或视频

以下是我们 Windows 风格指南中的一些提示：

### <a name="1-be-helpful"></a>1.有所帮助

- 避免使用营销风格或语言。
- 当演示或建议某事时，确保最终结果简洁明了；如果客户不知道这么操作的原因，那么只向他们展示操作方法是没有效果的。
- 不显示客户不需要的提示。

### <a name="2-show-dont-tell"></a>2.示人

尽可能简化文本（考虑小型动画或视频）。

### <a name="3-dont-overwhelm"></a>3.不会严重影响

- 每个组合使用情况会话的弹出窗口和提示最多为 4 个，包括系统通知和 shell 通知。
- 确保弹出窗口的出现时机合适。
- 不要阻止客户执行操作。
- 确保弹出窗口可以轻松消除。

### <a name="4-keep-it-contextual"></a>4.使其保持上下文

- 教学片段在恰当时间显示最为有效。
- 创建教程或幻灯片时需确保信息具体明确。
- 拒绝营销“垃圾”- 专注于具体提示和技巧。
- 如果有必要，为客户提供之后返回教程的方法（安装说明可能仅相关一次，而人们第一次通常不会保留信息）。
- 空页面消息应该能让用户了解消息出现的原因并且/或者让用户感到舒心，因此应确保消息简洁实用。

### <a name="5-minimize-painful-setup"></a>5.最大程度减少令人头痛的安装程序

客户为了体验全部价值所需执行的其他操作（如登录到在线服务）应尽可能容易。

- 消息应该简短直接。
- 避免客户流失。 如果可能，请提供一种从客户所在地进行连接的方法。
- 如果可以，提供稍后执行的选项，然后提醒客户稍后执行。
- 如果在客户体验中略过了一些选项，应提供快速简单的切回方法。

## <a name="help-links"></a>帮助链接

以下是我们 Windows 风格指南中的一些提示：

### <a name="when-should-we-provide-a-help-link"></a>何时我们应提供的帮助链接？

几乎永远不会。 提供的帮助链接时，才：

- 还有一个明显且重要的问题的客户很可能会有，而不是在 UI 中的答案将帮助他们在 UI 任务成功完成。 
- 在 UI 中提供用户在 UI 任务成功执行所需的信息的数量没有足够的空间。 

### <a name="where-should-help-links-appear"></a>其中应帮助链接出现？ 

- 文本链接应显示尽可能帮助尽可能定向的 UI 元素。 
- 如果必须提供适用于整个 UI 屏幕的文本链接，将其放置在屏幕的左下角。 
- 如果提供的链接通过帮助按钮 （？），工具提示应为"帮助"。

### <a name="what-url-should-we-use"></a>我们应使用哪个 URL？

永远不会直接链接到 web 地址，而是使用重定向服务。

Microsoft 开发人员应使用 FWLink，除非它是用户可能需要手动键入，可使用该事例 aka.ms 链接 （只要 URL 的目标是自动识别的浏览器区域设置，如 Docs.microsoft.com 网站） 的帮助链接

### <a name="text-guidelines"></a>文本指南 

- 使用完整的句子。
- 不包括结束标点问号除外。 
- 无需为任务的标题; 使用相同的文本使用 UI 的上下文中有意义的文本，但请确保这两个之间的逻辑连接。 例如： 
- 帮助链接：如果允许异常的风险是什么？ 
- 帮助主题标题："允许程序通过 Windows 防火墙进行通信"
- 应尽可能具体地了解帮助主题的内容。 
    - 我们样式
        - Windows 防火墙如何帮助保护我的计算机？
        - 为什么突出显示可以提高图片
    - 不是我们的样式
        - 有关 Windows 防火墙的详细信息
        - 了解有关颜色管理的详细信息
        - 了解详细信息
- 使用整个句子的链接文本，而不仅仅是关键字。 
    - 我们样式 
        - [如果允许异常的风险是什么？]()
    - 不是我们的样式
        - 是什么[允许例外的风险]()？ 

## <a name="error-messages"></a>错误消息

下面是改编自 Windows 风格指南的一些指导：

编写了正确的消息是提供足够的说明，但却过于技术; 之间的平衡之间的休闲和又备受青睐，但不是令人讨厌或冒犯性内容。

### <a name="general-guidelines"></a>一般指南

使用每个错误情况的一条消息。

#### <a name="headings"></a>标题

- 保持简短并简要说明的问题是或**理想情况下要执行的操作**。 <br>某些 UI 图面可能有标题而不是包装在太长时截断，因此请留意这些。
- 在标题中使用解决方案，如果它是一个简单的步骤。
- 请确保直接与按钮相关标题如果读取器将忽略正文文本。
- 尽量避免使用"问题"中没有标题，除非您别无选择。 要更具体有关的问题。
- 在标题中，应避免使用变量 （如文件、 文件夹和应用程序名称）。 将它们放在正文中。

#### <a name="body"></a>正文

- 如果标题充分说明的问题或解决方案，则不需要正文文本。
- 不重复的消息的略有不同的用词的标题。
- 通信清晰和简洁的解决方案是什么。
- 重点放在第一次提供事实数据。
- 不要责怪用户错误。
- 如果没有错误代码与错误关联并且如果您认为包含错误代码可能会帮助客户或 Microsoft 支持部门来调研该问题，包括正下方的正文文本，并将其写入，如下所示：

    错误代码: # # #

    如果客户具有解决不使用代码错误所需的所有信息，不需要包含它。

#### <a name="buttons"></a>按钮

- 编写按钮文本，使它是对主要说明的特定响应。 如果这不可能，使用"关闭"上诉按钮文本 （而不是"正常"或"完成"）。
- 如果有多个按钮，将最左侧的按钮建议用户采取的操作。 使最右边的按钮更保守的操作，如"取消"。

#### <a name="help-links"></a>帮助链接

只考虑不能进行特定且可操作的错误消息的帮助链接。

## <a name="null-state-text"></a>Null 状态文本

下面是一些帮助从 Windows 样式指南。

Null 状态时发生的客户数据或内容不存在从应用或功能之后搜索，或在需要时，不返回任何结果时缺少的信息从窗体，例如计费事务的信息。

### <a name="guidelines"></a>指南

 - 如果可能，这个机会，将使用 null 状态的情况下要向他人传授关于如何使用功能 （例如，如何添加音乐，何处进行查找图片，等等。）  
- 如果在 UI 中有一个标题，解释要采取的"修复"null 状态 （例如，"添加一些音乐"） 的操作 
- 文本的乐趣。 此空间可以是提供兴致勃勃，因为它可能不会看到几次的机会。 
- 避免"It's 孤独在此处。" 这是很遗憾，被过度使用。 
- 避免问题，例如"尚未连接您的打印机？" 好的用于一次，但此格式往往获取过度使用，并且问题给客户额外负担/带来压力。 它可以感觉表现优劣的用。 
- Null 状态文本中的各种是件好事。 

### <a name="examples"></a>示例

- "将某人添加为收藏项，并将在此处看到它们。"
- "您有任何成就或游戏剪辑感到特别自豪吗？ 将它们添加到你展示。"
- "没有人的参与方尚未中。 启动一台 ！"
- "当有人将你添加为友元时，您可以看到它们。"
- "执行操作如解锁成就，记录游戏剪辑并添加好友，您将看到它全部。"
- "您最喜欢的朋友会显示在此处，以便您可以看到它们何时联机以及它们是最多。"

## <a name="punctuation"></a>标点符号

- 标题或不完整的句子不加结束标点符号（句号、问号）。 标题包含问题的确认对话框例外
- 使用《Microsoft 风格指南》中关于[句号](https://docs.microsoft.com/style-guide/punctuation/periods)和[问号](https://docs.microsoft.com/style-guide/punctuation/question-marks)的指南。

## <a name="status-messages"></a>状态消息

状态消息由弹出 (toast) 消息和通知组成。

|字符串类型         | 说明                               |
|------------        |-------------------------------------|
|toast               |带有结束标点符号且首字母大写 - 最好带有对象变量，以便用户能在离开对象后了解消息适用于哪个对象。|
|通知标题|没有结束标点符号且首字母大写（这是一个标题） - 最好带有对象变量|
|通知详情|完整句子，最好带有显示对象的 UI 链接|

下面是通知消息的一些详细建议：

|字符串类型         | 说明                               |
|------------        |-------------------------------------|
|已启动             |尽可能省略 - 通常可以直接跳到正在进行的消息以减少干扰次数。|
|正在进行中         |开头是表示正在执行的操作的动词，结尾是表示操作正在进行的省略号。 以下是一个示例：<br> *创建卷"客户数据"...*|
|成功             |开头是“已成功”，结尾是软件刚刚完成的操作。 以下是一个示例：<br> *已成功创建卷"客户数据"。*|
|失败             |开头是“无法”，结尾是软件无法完成的操作。 以下是一个示例：<br> *无法创建卷"客户数据"。*|

## <a name="tooltips"></a>工具提示

很好的工具提示简要介绍未标记的控件，或在这种有用的带标签控件，提供少量的其他信息。 它们还可帮助客户通过提供其他导航 UI，没有重复 — 控件标签、 图标、 链接等有关的信息。

尽量少或根本不应使用的工具提示。 它们可以是客户，因此，不包括一个工具提示，只需重复标签或状态明显中断。 它始终应添加有价值的信息。

|    上下文                                 |    如何编写工具提示    |
|    -----------------------                 |    -------------------------    |
|当控件或 UI 元素是未标记...|使用简单的描述性名词短语。 例如：<br> 荧光笔 |
|当标记的 UI 元素时，但其目的需要澄清...|<ul><li>简要介绍如何使用此 UI 元素。 </li><li>使用命令性谓词窗体。 例如，"查找文本在此文件中"（不"来查找的文本在此文件中"）。</li><li>除非有多个完整的句子，不包括结束标点。</li> </ul>|
|当文本标签是被截断或可能在某些语言中截断的...|<ul><li>提供工具提示中未截断的标签。</li><li>可选：在另一个行中，根据需要提供澄清的说明，但仅。</li><li>如果未截断的信息的页或流上提供了其他位置并不提供工具提示。</li></ul>|
|如果键盘快捷方式位于...|<ul><li>可选：提供在后面的标签或说明性短语，例如括号中的键盘快捷方式"打印 (Ctrl + P)"或者"(Ctrl + F) 此文件中查找文本"</li><li>它是确定以将有用的键盘快捷方式添加到澄清工具提示，但避免添加工具提示，仅以显示键盘快捷方式。 </li></ul>|
